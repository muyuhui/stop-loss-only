## Context

该 change 是拆分路线的最后阶段，依赖核心监控、领域 API 和新版设置界面稳定。通知、导入和维护任务都必须是可关闭的附属能力，不能成为止损主事务依赖。

## Goals / Non-Goals

**Goals:**

- 提供隐私默认安全的系统通知和 Webhook。
- 提供可预览、可验证且不丢精度的数据导入导出。
- 控制历史和诊断数据增长并强化备份恢复。

**Non-Goals:**

- 不支持邮件、企业微信、券商专有 CSV 或公网部署。
- 不允许浏览器直接恢复活动数据库。
- 不让投递成功影响触发提交结果。

## Decisions

### 1. 站内告警是唯一事实来源

主事务提交后创建 `DeliveryAttempt`。Webhook 使用稳定幂等键、时间戳、HMAC 签名、硬超时和有限重试；投递失败只改变 attempt 状态。

### 2. 系统通知由用户明确授权

浏览器权限只在点击启用时请求。拒绝或不支持时保持完整站内告警。浏览器关闭期间不承诺系统通知送达。

### 3. Webhook 默认最小化和 SSRF 安全

只允许 HTTPS；阻止环回、私网和链接本地地址。Windows 本地密钥优先使用 DPAPI 封装的本机存储，API 只返回是否已配置。

### 4. CSV 使用两阶段提交

预览解析不写数据库，返回短期本地令牌和逐行结果；确认时重新校验令牌并原子提交。导出数字为文本语义，所有公式前缀转义。

### 5. 维护任务有界且可验证

历史分批清理并保留可恢复进度；图表服务端下采样已经由前置仓位领域 change 提供，本 change 不重复实现。备份使用 SQLite 一致性快照，manifest 包含 checksum、schema 版本和 WAL 感知信息；UI 只能创建备份，恢复仍需停服命令。

## Risks / Trade-offs

- [Webhook 泄露持仓] → 默认关闭、最小 payload、显式详情级别和脱敏日志。
- [系统通知不可靠] → 明确其为尽力投递，站内告警始终权威。
- [CSV 注入或超大文件耗尽资源] → 限制大小/行数/编码并转义公式前缀。
- [历史清理持有 SQLite 写锁] → 小批量、可恢复游标和 busy timeout。

## Migration Plan

1. 先增加配置与投递表，所有通道默认关闭。
2. 上线系统通知，再上线 Webhook；分别通过隔离测试。
3. 上线 CSV 预览/提交和安全导出。
4. 最后启用保留清理、诊断包和 UI 备份入口。
5. 任一扩展可独立关闭；回滚保留投递与审计记录，不影响核心表。

## Open Questions
