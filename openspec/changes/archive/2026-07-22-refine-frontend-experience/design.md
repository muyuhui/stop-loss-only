## Context

前端采用 Vue 3、Element Plus、Pinia 和 Vue Router，当前各页面直接组合 Element Plus 组件，并在模板中使用大量行内样式。业务流程已经完整，但缺少共享视觉变量、统一页面状态和明确响应式策略：桌面导航会提前折叠，390px 视口出现页面级横向滚动，摘要卡被压缩为窄列，宽表格只显示部分字段；详情页还存在信息重复和危险操作层级不清的问题。

本次改造横跨应用外壳、仪表盘、持仓、告警、设置和表单，但继续沿用现有数据模型与 API。视觉目标不是增加装饰，而是形成安静、风险优先、长时间使用不疲劳的监控界面。

## Goals / Non-Goals

**Goals:**

- 建立统一且克制的颜色、排版、间距、面板和状态表达体系。
- 在 360px 及以上常见视口中消除页面级横向滚动，保持核心任务完整可操作。
- 让仪表盘首先回答是否存在风险、哪些持仓接近止损以及数据是否新鲜。
- 降低持仓列表和详情的信息重复，缩短新增、查看、修改与平仓路径。
- 为加载、刷新、失败、空数据和数据过期提供一致反馈。
- 改善图标按钮、键盘焦点、状态文本与危险操作的可访问性。

**Non-Goals:**

- 不修改后端 API、数据库结构、止损算法或行情来源。
- 不引入图表库、大型设计系统、主题市场或复杂动画。
- 不实现暗色主题、用户自定义配色或可拖拽仪表盘。
- 不将本项目扩展为专业交易终端，也不增加交易执行能力。

## Decisions

### 1. 使用语义化设计变量，而不是逐页微调行内样式

新增全局样式入口，以 CSS 自定义属性定义背景、表面、文本、弱文本、边框、品牌、盈利、安全、警告、危险、圆角、阴影和间距。页面只使用语义类名与少量局部样式，Element Plus 通过 CSS 变量覆盖保持一致。

选择这一方式是因为它能在不引入新依赖的情况下统一现有组件，并让后续调整集中发生。备选方案是引入新的组件库或原子 CSS 框架，但会增加迁移成本、包体积和双设计系统并存的风险。

### 2. 采用三段式响应式结构

- `>= 1024px`：完整顶部导航、四列摘要、桌面表格和多列详情。
- `768px–1023px`：紧凑顶部导航、两列摘要、可压缩表格或重点列布局。
- `< 768px`：紧凑品牌栏配底部主导航、两列摘要、持仓卡片、单列表单和纵向操作区。

响应式切换由 CSS 媒体查询和语义容器完成，不使用 User-Agent 判断。桌面表格与移动卡片共享同一份数据及格式化函数，只改变展示结构。页面最小支持宽度为 360px，并禁止应用外壳产生横向滚动；仅确有必要的局部数据区域可以显式滚动且必须有视觉提示。

### 3. 仪表盘采用“风险 → 资产 → 明细”层级

首屏先展示组合风险摘要和需要处理的告警，再展示总盈亏、市值与成本等资产指标，最后按距止损从近到远呈现活动持仓。无告警时使用紧凑状态行，不保留大面积空卡片。金额使用等宽数字特性，风险颜色只用于需要注意的状态。

备选方案是保留四张等权卡片并只调整样式，但无法解决用户第一眼找不到风险重点的问题。

### 4. 持仓列表和详情围绕任务合并信息

桌面列表将名称与代码、当前价与盈亏、止损价与距离组合展示；整行提供明确的详情入口。移动端使用持仓卡片，完整保留核心价格、盈亏、止损距离和状态，不依赖横向表格。

详情页顶部集中展示当前价、盈亏、止损价、距止损和生命周期状态；价格信息与止损设置不再重复。刷新动作在当前页真实执行，不能再以“手动刷新价格”的名称跳转设置页。平仓属于业务动作，删除进入独立危险区域。

### 5. 使用共享的异步页面状态模型

数据页统一区分：首次加载、已有数据刷新、成功、空数据、可恢复错误和数据过期。首次加载显示骨架或明确加载状态；后台刷新保留现有内容并显示轻量进度；失败时保留最后成功数据、显示失败信息和重试入口；成功后记录最后更新时间。

这一模型可由轻量组合函数或各页一致的状态字段实现，不引入新的状态管理依赖。仪表盘继续遵守单一轮询计时器约束。

### 6. 设置页提供预设并保留高级控制

默认显示三个易理解预设：`及时`（10 秒/1 分钟）、`均衡`（30 秒/5 分钟）和 `省资源`（60 秒/15 分钟），分别映射到现有页面轮询间隔与后端监控间隔。当前值不匹配预设时显示“自定义”。原有两个数值输入放入可展开的高级设置，范围和 API 字段不变。

这样可降低普通用户理解成本，同时不损失已有控制能力。立即刷新价格移动到更符合上下文的仪表盘或持仓区域，设置页可保留入口但不作为主要操作。

### 7. 以内容和结构保证可访问性

所有仅图标按钮必须有可访问名称；键盘焦点必须清晰；盈利、亏损、告警和状态除颜色外还需通过符号、文本或图标表达；中文界面不混用无必要的英文控件文案。危险操作保持二次确认，并在视觉上与常规操作隔离。

## Risks / Trade-offs

- [同时维护桌面表格和移动卡片可能产生展示逻辑重复] → 抽取共享格式化函数和小型字段组件，数据请求与业务判断只保留一份。
- [全局覆盖 Element Plus 变量可能影响未检查组件] → 只覆盖已使用的语义变量，逐页执行视觉回归和键盘检查。
- [风险优先排序会改变用户熟悉的持仓顺序] → 明确标注排序依据，并在持仓管理页继续保持稳定分页顺序。
- [显示数据过期状态需要定义阈值] → 以成功轮询间隔的两倍作为前端过期提示阈值，不改变后端行情新鲜度规则。
- [底部导航占用移动端垂直空间] → 仅保留四个主入口并处理安全区，页面内容增加对应底部留白。
- [设置预设可能掩盖精确控制] → 高级设置始终可达，并在自定义值生效时明确显示“自定义”。

## Migration Plan

1. 先建立视觉变量、应用外壳和共享状态组件，保持现有页面内容可用。
2. 依次迁移仪表盘、持仓列表、持仓详情与表单，再迁移告警和设置页。
3. 为桌面、平板和 360–390px 窄屏补充结构测试与手动视觉检查。
4. 保留现有 API 与路由，因此发布无需数据迁移；若视觉回归失败，可按页面回退组件而不影响后端。
