## Context

现有 API 直接围绕 ORM 和路由内临时计算构建，导致同一持仓在新建、详情、列表和仪表盘中表现不同；分页、状态码和设置也与规格不一致。本设计假定前一变更已经落地定点金额以及 `holding`、`triggered`、`closed` 生命周期。

## Goals / Non-Goals

**Goals:**

- 建立可复用、强类型的响应契约。
- 同一持仓快照在所有接口返回一致的派生字段。
- 让分页、筛选、删除、仪表盘、刷新和设置行为与 OpenSpec 一致。
- 服务重启后恢复监控间隔，前端保存设置后立即调整轮询。
- 用集成测试和 OpenAPI 校验外部契约。

**Non-Goals:**

- 不重复改造前一变更处理的行情适配器和触发幂等逻辑。
- 不加入认证、多用户、备份自动化或部署打包。
- 不引入手续费等新的投资分析能力。

## Decisions

1. **独立响应构建层。** 服务返回领域结果，统一构建器生成 Pydantic 响应；读取接口不再通过修改 ORM 临时“修复”派生字段。
2. **统一分页信封。** 集合接口返回 `{items, total, page, size}`，页码从 1 开始，size 有上限，排序固定为 `created_at DESC, id DESC`。
3. **写入边界持久化派生值。** 新建、修改止损、更新最高价和迁移时计算并保存止损价；读取仅验证一致性，不静默修改数据库。
4. **生命周期感知的仪表盘。** `holding` 和 `triggered` 计入当前敞口及未实现收益；`closed` 计入已实现收益；分别返回各状态数量。
5. **事务化设置服务。** 先校验完整候选设置，再应用运行时变更，成功后才提交数据库；启动调度器前读取已保存值。前端轮询器监听设置变化并重启唯一计时器。
6. **明确删除语义。** 成功删除返回空响应 204；告警依赖自己的快照保留历史，不级联删除审计记录。
7. **结构化刷新结果。** 响应包含周期 ID、交易状态、聚合数量、逐标的结果和新触发摘要；部分行情失败仍可返回成功周期，但必须逐项说明。

## Risks / Trade-offs

- **响应变化破坏现有前端** → 在同一变更内同步 schema、前端和契约测试，仅在必要时提供临时别名。
- **运行时设置应用可能失败** → 先应用调度器，再提交数据库；失败时恢复旧间隔。
- **持久化派生值可能漂移** → 提供显式不变量检查器，不在读取时静默重算。
- **新仪表盘口径可能令用户困惑** → UI 明确标注当前敞口、未实现收益和已实现收益。

## Migration Plan

1. 建立响应 schema、统一构建器和不变量测试。
2. 改造设置校验、启动恢复和运行时应用。
3. 切换持仓及刷新接口，再同步前端调用方。
4. 切换仪表盘聚合和展示标签。
5. 校验并发布 OpenAPI，验证后移除临时兼容别名。

## Open Questions

- 已实现收益当前是否明确为“未计手续费和税费的毛收益”？
