## Why

现有界面强调金额汇总，却没有把监控可信度、行情新鲜度、风险金额和待处置事项放在首位；筛选、分页和触发后操作也不完整。在新仓位领域 API 稳定后，需要围绕“可信度与行动”重做核心用户工作流。

## What Changes

- 重构仪表盘，优先显示监控状态、行情覆盖、触发风险和预计止损损失。
- 用新版 positions API 实现可搜索、筛选、排序、分页并可恢复上下文的仓位列表。
- 重做仓位详情，支持批次、部分平仓、风险确认、重新布防、事件时间线和关闭复盘模式。
- 重做告警列表，分离已读和已处置，提供真实分页、筛选与关联处置入口。
- 建立统一风险/监控/盈亏视觉语义、Decimal 安全格式化和响应式可访问组件。
- 集中 Pinia 服务端状态与轮询，保留最后成功数据并避免重复定时器。
- 增加真实挂载组件测试和三个目标视口的浏览器 E2E。
- 不在本变更中增加 Webhook、CSV 导入导出或修改领域核算公式。

### Modified Capabilities

- `dashboard`: 增加监控可信度、估值覆盖、仓位权重、风险金额和风险优先布局。
- `holdings-crud`: 使用新版仓位列表/详情工作流展示批次、行情状态、处置和关闭复盘。
- `alert-system`: 增加处置筛选、稳定分页、事件关联和移动端告警卡片。
- `frontend-shell`: 增加统一筛选、状态语义、财务格式化、响应式导航和可访问性要求。
- `runtime-settings`: 按监控、行情与历史等责任分组，并展示实际生效值和调度状态。
- `quality-gates`: 使用真实组件挂载和浏览器 E2E 覆盖核心风险工作流。

## Impact

- Vue 页面、组件、Pinia store、路由 query、轮询工具、样式与前端测试栈。
- 消费前两个 change 提供的监控和仓位 API，不新增数据库领域概念。
- 目标视口为 390x844、768x1024 和 1440x900，最低仍支持 360px 宽度。
