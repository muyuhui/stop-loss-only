## Context

系统在 FastAPI 进程内运行 APScheduler，通过 akshare 获取行情，使用 SQLite 浮点数字段保存价格，并在越过止损线时直接把持仓改为 `stopped_out`。定时刷新与手动刷新各自实现了一套相似逻辑，面对重复代码、行情失败、旧数据、并发刷新和进程重启时难以保证一致性。

## Goals / Non-Goals

**Goals:**

- 定时与手动刷新共用一个监控周期服务。
- 每轮对每类市场数据最多抓取一次，并按 `(asset_type, code)` 分发到全部匹配持仓。
- 只有成功且满足新鲜度要求的行情才能更新最高价和触发止损。
- 将阈值触发与用户确认平仓分离，并保证告警幂等。
- 统一十进制存储、舍入和比较规则，删除持仓后仍保留可读告警历史。

**Non-Goals:**

- 不连接券商、不自动下单，也不把告警描述成已成交。
- 不扩展到当前 A 股和基金范围之外的市场。
- 不建设分布式任务平台或多用户系统。
- 不在本变更中全面重构 API 响应格式。

## Decisions

1. **统一监控周期服务。** 服务负责读取活动持仓、批量取价、校验新鲜度、更新最高价、判断阈值和创建告警；调度器与 `/prices/refresh` 仅调用它，避免双实现漂移。
2. **统一时间模型。** 交易日统一为 `date`；交易时段使用 Asia/Shanghai 时区，默认上午 09:30–11:30、下午 13:00–15:00。交易日历失败时可退化到工作日规则，但必须记录并返回 degraded 状态。
3. **按数据集批量取价。** 每轮按需各读取一次股票、ETF、LOF 快照；场外基金按唯一代码获取。结果以 `(类型, 代码)` 建索引，同代码多笔持仓都能得到更新。
4. **新鲜度是触发前置条件。** 规范化行情包含 `quoted_at`、`fetched_at`、`source` 和 freshness。失败或过期行情只能用于错误展示，不得更新最高价或新建触发。
5. **采用 `holding → triggered → closed`。** 越线后原子进入 `triggered` 并创建告警；用户确认成交后记录实际平仓价并进入 `closed`。历史 `stopped_out` 迁移为 `closed`，避免猜测历史意图。
6. **数据库兜底幂等。** 告警保存代码、名称及行情快照，并使用生命周期键和唯一约束阻止重复活动告警；应用层重试不能绕过数据库约束。
7. **定点精度。** 金额字段使用 `Numeric/Decimal`，中间计算保留足够精度，仅在持久化或展示边界舍入。

## Risks / Trade-offs

- **akshare 字段可能变化** → 用适配器隔离字段映射，并以代表性 DataFrame fixture 做契约测试。
- **交易日历退化可能把节假日视为交易日** → 显式标记 degraded，且仍强制交易时段限制。
- **新增状态会影响现有前端** → 迁移期提供明确映射，并同步更新状态展示与操作。
- **SQLite 并发能力有限** → 缩短事务，用唯一约束处理重复周期。
- **浮点迁移可能产生舍入** → 迁移前备份，声明精度，并校验行数及代表性汇总值。

## Migration Plan

1. 新增定点字段、行情元数据、告警快照、生命周期键与索引。
2. 备份数据库，回填金额和状态，校验行数及汇总值。
3. 上线行情适配器与统一监控服务，迁移期保留旧响应字段。
4. 清理重复告警后启用唯一约束。
5. 将调度器和手动刷新切换到统一服务，再移除旧循环与浮点字段。
6. 回滚时恢复迁移前备份和上一版应用。
