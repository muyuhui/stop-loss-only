## Why

单一 `Holding` 同时承担标的、交易批次、仓位、止损规则、行情和生命周期，无法可靠支持加仓、部分平仓、费用税费、风险处置和历史审计。监控可信性稳定后，需要通过可对账的影子迁移引入清晰领域模型。

## What Changes

- 分离标的、账户、仓位、交易批次、平仓分配、版本化止损规则和不可变仓位事件。
- 将仓位生命周期、风险处置状态和“是否发生部分平仓”拆为正交事实，避免组合状态冲突。
- 使用 Decimal 和默认 FIFO 实现多批次、部分/全部平仓、费用税费及已实现/未实现盈亏。
- 支持风险确认和提交新规则的重新布防，保留原触发与告警快照。
- 引入有序迁移、影子读取、对账和兼容 DTO；明确新旧模型的单一事实来源与切换点。
- 新增版本化仓位 API，同时在一个明确发布周期内保留现有 holdings 路由。
- 不在本变更中重做全部前端页面、增加 Webhook 或 CSV 导入导出。

## Capabilities

### New Capabilities

- `portfolio-accounting`: 交易批次、平仓分配、Decimal 核算、估值覆盖率和组合风险指标。
- `position-history`: 不可变仓位事件、规则版本和有界行情/止损线历史查询基础。

### Modified Capabilities

- `holdings-crud`: 支持加仓、部分/全部平仓、正交状态和关闭仓位复盘数据。
- `stop-loss-engine`: 支持版本化规则、风险确认、重新布防和新的触发序列。
- `alert-system`: 将告警阅读状态与风险处置状态分离，并关联不可变事件。
- `dashboard`: 使用新模型返回 Decimal 安全且带估值覆盖率的组合汇总。
- `local-operations`: 使用有序迁移、迁移对账、可验证备份和受控回滚。

## Impact

- 后端 ORM、迁移器、领域服务、持仓/告警/仪表盘 API 和测试 fixture。
- SQLite 新增领域表、索引和迁移版本；旧表在兼容期保留。
- 需要在代表性旧数据库上验证升级、对账、中断恢复和声明支持的降级。
- 前端继续可通过兼容 DTO 工作，新版页面由后续 change 交付。
