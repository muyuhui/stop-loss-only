## Context

该 change 依赖监控可信 API 和新仓位 API 已稳定。它只改变信息架构、交互和前端状态管理，不重新定义核算公式或数据库状态。

## Goals / Non-Goals

**Goals:**

- 风险和数据可信度优先于装饰性汇总。
- 桌面高密度操作与移动端核心处置都高效可访问。
- 列表筛选、分页和返回上下文稳定。
- 后台失败保留最后成功数据并显示年龄。

**Non-Goals:**

- 不增加新的领域状态、通知通道或数据导入能力。
- 不在浏览器计算财务结果。
- 不引入大型服务端状态框架。

## Decisions

### 1. 四入口信息架构

保持“仪表盘、持仓、告警、设置”，历史放在仓位详情。监控状态条与业务风险分别表达，避免健康绿色与盈亏/危险颜色混淆。

### 2. URL 保存列表上下文

搜索、筛选、排序和页码写入 query。桌面使用稳定列宽表格，移动端使用紧凑可展开卡片；关键行情状态和止损距离不可隐藏。

### 3. Pinia 集中服务端状态

按 monitoring、positions、alerts、settings 划分 store。应用外壳拥有唯一轻量轮询；页面订阅而不创建重复定时器。后台失败保留最后成功值，手动刷新用 cycle ID 显示进度。

### 4. 状态不能只靠颜色

中国市场盈亏继续红盈绿亏，但必须带正负号；危险、降级和未知分别使用图标、文本和结构。财务值使用 tabular numbers 和共享 Decimal 字符串格式化。

### 5. 测试可观察行为

工具函数单测之外，真实挂载页面验证加载、错误、筛选、表单和状态保留；浏览器 E2E 覆盖创建、刷新、触发、确认、重新布防、部分平仓和关闭。

## Risks / Trade-offs

- [页面信息密度过高] → 风险优先、渐进披露和可隐藏次要列。
- [移动端操作被固定导航遮挡] → 使用安全区和内容底部留白，并纳入 E2E 截图检查。
- [轮询重复或竞态] → 单资源单在途请求和单计时器所有权。
- [新 API 错误导致旧数据闪空] → 区分首次加载与后台刷新，后者保留最后成功数据。

## Migration Plan

1. 先建立共享格式化、状态组件、筛选工具栏和 store。
2. 按仪表盘、持仓、详情、告警、设置顺序迁移页面。
3. 每页完成真实组件测试和三个目标视口检查后再切换路由。
4. 旧页面在本 change 内保留到核心 E2E 通过；回滚只需恢复旧路由入口。
